# 软件、版本和文件要求 🖥️

下面不是“必须全都会”的清单。缺什么，Skill 会告诉你什么时候需要。

## 最常用的软件

| 软件 | 用来干什么 | 新手要不要先装 |
|---|---|---|
| Codex 桌面端 | 对话、整理需求、运行 Skill | 需要 |
| Blockbench 桌面稳定版 | 建模、贴图、骨骼、动画 | 做模型时需要 |
| Java JDK | 编译 Minecraft Java Mod | 只有做 Mod 时需要 |
| 对应 Mod 工具链 | NeoForge、Forge 或 Fabric 项目 | 只有接入游戏时需要 |
| Python 3 | 运行模型和交付检查脚本 | 推荐 |

## Windows 推荐做法

- 使用 PowerShell 7；
- 文件统一保存为 UTF-8；
- 路径尽量用英文和数字；
- 不要把项目直接放桌面、下载目录或网盘根目录；
- 每个模型单独一个文件夹；
- 修改前保留 Git 提交或备份。

推荐路径：

~~~text
D:/MinecraftProjects/fangjie-assets/crystal_tower/
~~~

不推荐：

~~~text
C:/Users/你的名字/Desktop/新建文件夹/最终版2/
~~~

## Minecraft 和 Mod 版本要先锁定

不要只说“Java 版”。至少要确认：

- Minecraft 版本；
- 使用 NeoForge、Forge 还是 Fabric；
- 对应加载器版本；
- JDK 版本；
- Blockbench 项目格式；
- 模型在游戏里是实体、方块、物品还是盔甲。

版本没定时可以先做概念图，但不适合直接做最终导出。

## Blockbench 项目要保持干净

- 每个骨骼和组都用看得懂的英文名；
- 转轴放在真正需要转动的位置；
- 不重复使用含义不明的 group1、cube23；
- 贴图尺寸和精度选择保持一致；
- 动画事件使用统一名称；
- 保存可继续编辑的源文件，不只留导出结果。

## 纹理不是越大越好

| 尺寸 | 建议 |
|---|---|
| 64×64 | 简单物品和小型方块模型的最低选择 |
| 128×128 | 普通模型的默认选择 |
| 256×256 | Boss、机械体、重点道具 |
| 512×512 | 大型展示、近距离镜头 |

高精度会带来更多绘制和修改时间，也会增加文件大小。如果模型只有几个简单方块，高精度没有明显意义。

## 动画最容易出问题的地方

1. **转轴放错**：想让部件绕水晶转，转轴却放在部件自己身上；
2. **方向相反**：左右对称部件用了相反角度；
3. **重叠**：两个部件走进同一位置；
4. **穿模**：关键帧没碰到，中间过渡帧却穿过去；
5. **立刻归位**：用户要 5 秒冷却，动画却攻击完马上收回；
6. **事件错位**：能量弹已经发出，音效和粒子还没触发。

所以检查不能只看几个关键帧，必须把完整动画从头到尾播放。

## 粒子和音效不是模型文件自动负责的

Blockbench 可以准备动画和事件，但真正让粒子、音效、伤害和能量弹在游戏中发生，通常还需要 Mod 代码。

大白话：

- 模型负责“长什么样、怎么动”；
- 纹理负责“表面是什么样”；
- 粒子负责“周围冒什么效果”；
- 音频负责“什么时候听到什么声音”；
- Mod 代码负责“这些东西在游戏里怎么一起工作”。

## 多模型项目怎么防止搞混？

每个模型使用唯一 asset_id，例如：

~~~text
mecha_cerberus
crystal_tower
arcane_greatsword
~~~

所有动画、声音、粒子和导出文件都带这个身份。每次绑定前显示当前模型名，发现名字不一致就停止。

## 版权和安全

- 只使用自己拥有或获准使用的参考图、模型、贴图和声音；
- 不把密码、令牌或私人文件放进项目；
- 上传 GitHub 前检查有没有密钥和个人路径；
- 不直接复制商业游戏的受保护模型；
- 可以描述“机械能量塔”这类风格方向，但最终造型应当有自己的设计。

## 什么时候才算完成？

不是“文件生成了”就算完成，而是：

- 能打开；
- 能看到贴图；
- 动画能完整播放；
- 没有明显穿模；
- 事件顺序正确；
- Mod 能编译和启动；
- 有截图、日志或报告证明这些结果。

环境准备好，流程就会稳定很多。Enjoy building! 🎮

Approval first · Build second

你说想法，Skill 把它变成能做的模型。

它不是听到“做个怪兽”就直接乱建模，而是先问清楚、给三套图、等你选中，再去 Blockbench 正式制作。✨

最强的地方不是“生成快”

真正重要的是少返工。外观、纹理、动画、粒子、音效、损毁方式和保存位置都会在生产前说清楚。你没明确批准，它就不能正式开工。

大白话：先把话说明白，再把活干漂亮。

A / B / C三套方案，选中才开工
✅ 必须遵守的确认门 “还可以”“差不多”都不算批准。只有“我选择 B，按 B 开始正式建模”这类明确说法才算。

Workflow

四步看懂整个过程

01

说想法

告诉它做什么、给谁用、想要什么感觉。

02

问清楚

确认版本、精度、动画、效果、配套物和保存位置。

03

选方案

先看 A、B、C 三套 Blockbench 可实现的图。

04

做和检查

建模、贴图、动画、接 Mod，再用证据验证。

Real examples

三个实际例子

点下面的按钮，看看用户怎么说，以及 Skill 需要继续检查什么。

机械三头狼 Boss

做一个黑铁机械三头狼，眼睛发红光。
要待机、走路、三个攻击、受伤、死亡和损毁动画。
死亡后留下冒烟残骸，纹理按 256×256 评估。

Skill 还要问：三个头是一起还是轮流攻击？红眼只发光还是还要粒子？残骸整体倒下还是零件散开？要不要 Boss 蛋、掉落核心和专属武器？

水晶能量塔

塔身里的能量从下往上输送。
攻击时两块护盾围着水晶中心转，水晶发射能量弹。
停火后继续转 5 秒，再慢慢归位。

Skill 还要检查：护盾是绕水晶公转，不是自己原地转；两块护盾方向一致、间距固定；完整动画中不能碰到水晶；启动、蓄能、发射、命中和冷却音效顺序正确。

三阶段魔法巨剑

基础状态是暗银巨剑。
蓄力时符文从下往上点亮。
满蓄力出现青蓝火焰，耐久归零切换断刃模型。

Skill 还要问：第一人称和第三人称比例怎么定？符文多久亮完？满蓄力声音什么时候播放？断裂位置在哪里？是否还需要展示架？

两块护盾围绕同一个中心公转，保持相同朝向和固定距离。旋转半径太小就会碰到水晶；只看关键帧正常，也不代表中间帧不会穿模。

Texture

纹理到底选多大？

尺寸适合你会得到什么
64×64小物品、简单方块文件小，但细节少
128×128大多数普通模型效果和工作量比较平衡
256×256Boss、机械体、重点武器能画磨损、接缝和发光细节
512×512大型近景展示最细，但制作和修改最慢
高像素不等于自动高质量 如果选了 512×512，材质变化、划痕、接缝、阴影和高光也必须跟着做到高水平。只放大画布没有意义。

Identity

一个模型，一个文件夹

这是防止模型、动画、音效和粒子互相串台的最简单办法。每个模型再加一个唯一的 asset_id，绑定前先对名字。

crystal_tower/
├─ model/       模型源文件
├─ textures/    纹理
├─ animations/  动画
├─ audio/       音效
├─ particles/   粒子
├─ previews/    预览图
├─ mod/         Mod 接入
└─ evidence/    截图、日志、检查报告

例如水晶塔使用 crystal_tower，机械三头狼使用 mecha_cerberus。任何文件身份不一致，就先停下来检查。

Delivery

什么才算真的完成？

  • 模型源文件能正常打开；
  • 贴图没有丢失或串到别的模型；
  • 完整动画播放时没有重叠和穿模；
  • 粒子、声音和伤害事件顺序正确；
  • Mod 能编译并进入游戏测试；
  • 有预览图、日志或报告证明这些结果。
🎮 最后记住 没问清楚，不建模；没看三套图，不选方案；没明确批准，不正式生产；没有验证证据，不算完成。
